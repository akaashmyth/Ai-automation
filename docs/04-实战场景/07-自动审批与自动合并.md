# 自动审批与自动合并的双 Automation 配置（方案 B）

> 实战场景：一个 Automation 负责「根据 requirements.md 实现需求并提 PR」，另一个 Automation 负责「自动审查并 Approve，触发 GitHub Auto-merge」

---

## 一、整体思路

- **Automation 1：Requirements Implementer**
  - 触发：定时（Scheduled），例如每小时
  - 职责：读取 `docs/requirements.md`，在仓库中实现/更新需求，并通过 **Open pull request** 创建 PR
- **Automation 2：Auto Approver**
  - 触发：GitHub PR 事件（`Pull request opened` 等）
  - 职责：审查 PR，满足安全规则时使用 **Comment on pull request** 工具提交 Approve 审查，从而触发 GitHub 的 Auto-merge

---

## 二、Automation 1：Requirements Implementer（实现需求）

### 1. 触发器

- 类型：**Scheduled**
- 示例：Every hour
- 仓库：`Ai-automation`
- 分支：`main`

### 2. 工具

- 勾选：**Open pull request**
- 可以不勾 Comment on pull request（只负责实现与提 PR）

### 3. Instructions 示例

```text
你是这个仓库的实现工程师。你的任务是根据 docs/requirements.md 中的需求，自动实现或更新代码，并通过 Pull Request 提交。

工作流程：

1. 读取 docs/requirements.md，理解当前实验用需求（尤其是最新的需求，比如 Banner 相关的 REQ-001 ~ REQ-005）。
2. 分析本次运行与上次运行相比，requirements.md 中有哪些新增或修改的需求条目。
3. 针对这些新增/修改的需求，在仓库中进行实现或更新，例如：
   - 在 index.html 中增加或调整对应的 HTML 结构
   - 在 css/style.css 中补充或修改样式
   - 如有需要，在 js/app.js 中增加交互逻辑
4. 代码风格尽量与现有文件保持一致，不要大改无关结构。
5. 完成实现后，自查：
   - 页面能正常打开（至少结构合理）
   - 样式大致符合 requirements.md 中的描述
6. 当有实质性改动时：
   - 创建一个新分支
   - 提交这些变更
   - 通过 Open pull request 工具打开一个新的 PR，标题格式类似：
     feat: implement requirements from docs/requirements.md
   - 在 PR 描述中简单列出本次实现的 REQ 编号列表（例如 REQ-001, REQ-002）。
7. 如果本次运行没有发现新的需求需要实现，则不创建 PR。

除非非常必要，不要删除现有功能代码。
```

---

## 三、Automation 2：Auto Approver（自动审批）

### 1. 触发器

- 类型：**GitHub**
- 事件：`Pull request opened`（或视情况加上 `Pull request pushed` / `CI completed`）
- 仓库：`Ai-automation`
- 范围：`by Anyone`

### 2. 工具

- 勾选：**Comment on pull request**
  - 在工具配置中选择 **Allow PR Approval**，允许自动 Approve
- 可以不勾 Open pull request（只负责审查，不创建 PR）

### 3. Instructions 示例

```text
你是这个仓库的自动代码审核与合并助手。你的目标是：在满足安全前提的情况下，自动审批（approve）Pull Request，从而触发 GitHub 的自动合并（auto-merge）。

工作范围：
- 只处理当前触发这个 Automation 的 Pull Request。
- 不负责自己写代码或创建新的 Pull Request，只做「审查 + 决定是否 approve」。

工作流程：

1. 读取本次 PR 的关键信息：
   - 变更的文件列表和 diff 内容
   - 作者（如果可见）
   - CI / Checks 状态（如果可用）

2. 进行快速安全审查，按以下规则判断是否可以自动批准：

   允许自动批准的条件（全部满足才通过）：
   - 文件范围仅限于以下类别：
     - 文档：docs/**, README.md
     - 前端静态页面和资源：index.html, list.html, detail.html, css/**, js/**
   - 不包含以下类型改动：
     - 删除大量代码块（例如整段函数、整个文件被删除）
     - 对敏感配置文件的修改（如 .env, package.json, 构建配置、CI 配置等）
   - diff 中不存在明显的危险模式：
     - 直接引入外部脚本的 `<script src="http://...">`
     - `eval(...)`、`new Function(...)` 等高风险 JS 用法
   - 如果有 CI / Checks 信息：
     - 所有必须的检查均为通过状态（绿色），没有失败或挂起过久的检查。

   如果任意一条不满足，则视为不安全，不进行自动 approve。

3. 决策逻辑：

   - 如果满足上述所有「允许自动批准的条件」：
     - 使用 Comment on pull request 工具，对当前 PR 提交一个 Approve 审查。
     - 在评论中简要说明：
       - 本次为自动审批
       - 审查范围（仅前端/文档）
       - 已检查的条件（文件范围、CI 状态等）
   - 否则：
     - 不做 Approve，只在 PR 中留下一个普通评论，说明：
       - 发现不满足自动合并策略的原因（例如改动范围包含配置文件，或 CI 未通过）
       - 请人工审查后决定是否合并。

4. 绝不要执行以下行为：
   - 不自行修改代码或推送新的 commit。
   - 不强制合并存在红色 CI 或高风险改动的 PR。
   - 对无法看清风险的复杂改动，一律不要自动批准。

5. 输出要求（在 PR 评论中）：
   - 若自动批准：
     - 说明「本次为自动审批，满足预设安全规则，将触发 GitHub 自动合并」。
     - 列出本次变更涉及的文件路径概览。
   - 若未批准：
     - 说明未自动批准的具体原因（例如：涉及配置文件 / CI 未通过 / 改动范围过大）。
```

## 四、与 GitHub Auto-merge 的配合

要让自动 Approve 真正触发「自动合并到 main」，需要在 GitHub 仓库中：

1. 打开仓库 Settings → Branches，启用 **Auto-merge**。
2. 在分支保护规则中设置：
   - 至少需要 1 个 Approving review。
   - （可选）需要 CI 检查通过。

这样，当 Automation 1 创建 PR、Automation 2 自动 Approve 且 CI 通过后，GitHub 会自动将 PR 合并到 `main`。
