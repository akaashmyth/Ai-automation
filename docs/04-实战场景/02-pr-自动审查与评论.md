# PR 打开时自动做代码审查的 Automation 配置

> 实战场景：当 PR 被打开时，自动进行代码审查并发表评论

---

## 一、创建 Automation

1. 前往 [cursor.com/automations](https://cursor.com/automations)
2. 点击 **Create automation**
3. 可选择模板或从空白开始

---

## 二、配置触发器

1. 在 **Triggers** 中选择 **GitHub**
2. 选择事件类型：**Pull request opened**（PR 打开时）
3. 选择要监听的**仓库**和**分支**（如 `main`）

> 注意：只有 GitHub 触发器才能使用「Comment on pull request」工具，所以必须选 GitHub 触发器。

---

## 三、启用工具

在 **Tools** 中勾选：

| 工具 | 用途 |
|------|------|
| **Comment on pull request** | 在 PR 中发表审查评论（必选） |
| **Read Slack channels** | 可选，如需将结果同步到 Slack |
| **Open pull request** | 可选，若希望 Agent 能直接改代码并提 PR |

---

## 四、编写 Prompt

在 **Instructions** 中写入审查规则，例如：

```
你是一个代码审查助手。当 PR 被打开时，请对本次变更进行审查。

审查范围：
1. 代码质量：命名、结构、可读性
2. 潜在 Bug：逻辑错误、边界条件、空指针
3. 安全：SQL 注入、XSS、敏感信息泄露
4. 性能：不必要的循环、N+1 查询
5. 测试：是否有对应测试

输出格式：
- 对每个问题，使用 inline 评论标注到具体代码行
- 严重问题用「必须修复」标记，建议用「建议」标记
- 若整体无重大问题，在 PR 顶部发一条总结评论说明审查通过

只对确实有问题的地方评论，避免过度审查。
```

可根据团队规范调整上述内容。

---

## 五、其他设置

| 设置 | 建议 |
|------|------|
| **Model** | 使用默认或更高级模型 |
| **Environment** | 若只读代码、不运行测试，可关闭；需要跑测试则开启 |
| **Permissions** | 按团队需求选择 Private / Team Visible / Team Owned |

---

## 六、保存并测试

1. 点击 **Save** 保存
2. 在目标仓库打开一个测试 PR
3. 等待 Automation 运行（通常几分钟内）
4. 在 PR 的评论中查看审查结果

---

## 流程示意

```
PR 打开 → GitHub 触发 Automation → Cloud Agent 拉取代码
    → 按 Prompt 审查 → 通过 Comment on PR 发表评论
```

---

## 常见问题

| 问题 | 处理方式 |
|------|----------|
| 没有收到评论 | 检查仓库是否已连接 GitHub、触发器是否选对 |
| 评论太泛 | 在 Prompt 中明确要检查的项和输出格式 |
| 运行太慢 | 大 PR 会耗时更长，可考虑在 Prompt 中限制审查范围 |
