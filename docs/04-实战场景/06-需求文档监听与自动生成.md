# 监听需求文档 + Push 触发自动写新需求

> 实战场景：当 master 分支有推送时，自动读取需求文档并生成新需求

---

## 一、触发器：监听 master 分支 Push

1. 在 Automation 的 **Triggers** 中选择 **GitHub**
2. 选择 **Push to branch**
3. 选择你的仓库，分支填 `master`（或 `main`）

每次向 master 推送代码时，Automation 都会运行。

---

## 二、读取需求文档

需求文档通常放在仓库中，例如：

- `docs/requirements.md`
- `requirements.md`
- `docs/需求文档.md`

Cloud Agent 会拉取仓库代码，可直接读取这些文件。在 Prompt 中明确指定路径即可：

```
请读取仓库中的 docs/requirements.md 作为需求文档。
```

---

## 三、「自动写新需求」的几种实现方式

| 方式 | 含义 | 需要的工具 |
|------|------|------------|
| **A. 更新需求文档** | 在现有文档中新增/修改需求内容 | Open pull request |
| **B. 创建 Linear Issue** | 把需求拆成 Linear 任务 | Linear 集成 + MCP 或相关工具 |
| **C. 创建 GitHub Issue** | 把需求拆成 GitHub Issue | 需要能创建 Issue 的工具（如 MCP） |
| **D. 生成新需求文件** | 根据文档内容生成新的需求说明文件 | Open pull request |

---

## 四、推荐配置（以「更新需求文档」为例）

假设：**检测到 master 有 Push → 读取需求文档 → 分析并生成新需求内容 → 用 PR 提交更新**

### 1. 触发器

- GitHub → **Push to branch** → 仓库 + `master`

### 2. 工具

- **Open pull request**（用于提交新需求变更）

### 3. Prompt 示例

```
你的任务是监听需求文档的变更，并在有更新时自动补充新需求。

1. 读取 docs/requirements.md（或你的实际路径）作为主需求文档
2. 分析本次 Push 的变更，尤其是需求相关文件的修改
3. 根据变更内容，推断并撰写新的需求条目
4. 将新需求追加到需求文档中，格式与现有文档保持一致
5. 创建新分支，提交变更，并打开 Pull Request

新需求格式示例：
- [需求ID] 需求标题
  - 描述：...
  - 优先级：高/中/低
  - 来源：基于 xxx 变更推断

若本次 Push 与需求无关（如仅修复 typo），则不做任何操作。
```

---

## 五、流程示意

```
master 有 Push → 触发 Automation → Agent 拉取代码
    → 读取需求文档 → 分析变更 → 生成新需求
    → 创建分支并提交 → 打开 PR
```

---

## 六、注意事项

| 项目 | 说明 |
|------|------|
| **Push 触发器** | 需要手动指定仓库和分支，无法从事件自动推断 |
| **需求文档路径** | 在 Prompt 中写清楚，避免 Agent 读错文件 |
| **避免循环** | 若 PR 合并会再次 Push 到 master，可能重复触发，可在 Prompt 中说明「忽略由本 Automation 产生的 PR 合并」 |

---

## 七、配置前需确认

1. **需求文档路径**：如 `docs/requirements.md`
2. **「写新需求」具体指**：更新文档 / 创建 Issue / 其他
3. **触发逻辑**：每次 Push 都分析，还是仅当需求文档被修改时才分析
